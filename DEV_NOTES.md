# 开发笔记：踩坑记录

## 掌心/掌背识别踩坑实录

### 背景

在视频中实时显示手部穴位时，需要区分**掌心朝向镜头**和**掌背朝向镜头**，以决定显示掌心专属穴位（如劳宫）还是掌背专属穴位（如外劳宫）。

### 踩坑 1：用"手指弯曲度"判断掌心掌背——不可靠

**现象**：手指弯曲时（如握拳），掌心穴位常在手背画面中显示。

**原因**：早期版本以**手指弯曲度**为主特征（40%权重），假设"手指弯=掌心、手指直=掌背"。但这是错误的——**手指可以在任何朝向下自由弯曲或伸直**：
- 掌心朝向镜头时可以握拳（手指弯曲）
- 掌背朝向镜头时也可以握拳（手指弯曲）
- 同样，两种朝向下都可以张开手指

**教训**：设计特征时必须以**不可变的解剖学差异**为基础，不能依赖可变的关节姿态。

### 踩坑 2：MCP 骨性凸起才是金标准

**原理**：食、中、无名、小指的根部骨节（MCP 关节，landmark 5/9/13/17）是**骨性凸起**，无论手指如何弯曲或伸直，其形状保持不变：

| 朝向 | MCP 特征 | 原因 |
|------|----------|------|
| **掌背** | MCP 骨节比手腕更靠近镜头（Z 值更小） | 骨性凸起天然外凸 |
| **掌心** | MCP 与掌面齐平，不比手腕更近 | 掌心面平坦 |

**计算方法**：`mcp_prominence = wrist_z - mean(mcp_z)`，正值越大越可能是掌背。

### 踩坑 3：阈值设计的浮点精度陷阱

**现象**：测试数据 `mcp_prominence = 0.030 - 0.010 = 0.020`，条件 `mcp_prominence < 0.020` 竟意外触发。

**原因**：浮点精度问题。`0.030 - 0.010` 在 IEEE 754 下实际值略小于 `0.020`。

**修复**：将阈值从 `0.020` 改用 `0.018`，或使用 `<=` 运算符避免临界值误判。

```python
# 错误写法
if spread > 0.85 and mcp_prominence < 0.020:
    return "palm"  # 浮点误差导致误触发

# 正确写法
if spread > 0.85 and mcp_prominence <= 0.018:
    return "palm"  # 留出安全边际
```

**教训**：临界阈值比较时必须考虑浮点误差，阈值与实际值之间至少留出 `1e-6` 级别的安全边际。

### 踩坑 4：特征归一化因子导致所有值被 clip 到上限

**现象**：测试时 `spread`（手指张开度）在所有场景下都被 clip 到 `1.0`，完全失去区分度。

**原因**：`total_angle` 原始范围约 `[0.3, 0.9]`，但归一化因子设为 `0.45`，导致绝大部分值除以 `0.45` 后超过 `1.0`。

```python
# 错误：归一化因子太小
spread = np.clip(total_angle / 0.45, 0.0, 1.0)  # 几乎总是 1.0

# 正确：用合理的分母
spread = np.clip(total_angle / 0.75, 0.0, 1.0)  # 保留区分度
```

**教训**：归一化因子的选取必须基于真实数据分布范围。新特征上线前，先 `print` 一批真实帧的原始值和归一化结果，确认有区分度。

### 踩坑 5：不确定时不应放行

**现象**：`_check_orientation` 检测结果为 `None`（侧向或不确定）时，旧代码返回 `True`（放行），导致掌心穴位在手背后仍然显示。

**原因**：设计者担心漏标（false negative）多于错标（false positive）。但用户的反馈是"**掌心的穴位在手背呈现时不该显示**"，说明**错标比漏标更影响体验**。

```python
# 旧代码：不确定时放行
def _check_orientation(...):
    hand_ori = self.detect_hand_orientation(hand_lms)
    if hand_ori is None:
        return True  # 侧面不确定时不拦截 → 错误！
    return hand_ori == required

# 新代码：不确定时不显示
def _check_orientation(...):
    hand_ori = self.detect_hand_orientation(hand_lms)
    if hand_ori is None:
        return False  # 宁可漏标也不错标
    return hand_ori == required
```

**教训**：产品决策上，**错标（false positive）的成本通常高于漏标（false negative）**，尤其当用户明确抱怨"不该显示的东西显示了"时。

### 最终方案：四特征决策树

重写后的 `detect_hand_orientation` 使用四个与手指姿态无关的特征：

| 特征 | 权重 | 原理 | 不受手指弯曲影响 |
|------|------|------|:---:|
| MCP 凸出度 | 金标准 | wrist_z - mean(mcp_z) | ✅ |
| MCP 连线弯曲度 | 辅助 | 手背时 MCP 连线呈拱桥弧 | ✅ |
| 四指 2D 张开度 | 辅助 | 掌心时手指自然撑开 | ✅ |
| 拇指外展度 | 备选 | 掌心时拇指更外展 | ✅ |

决策树 6 层，以 MCP 凸出度为金标准：
- `prom > 0.015`（明显凸出）→ 掌背
- `prom < -0.010` → 掌心
- 微弱信号时交叉验证张开度和弯曲度
- 所有特征均模糊 → 返回 `None`

### 经验总结

1. **选特征优先选不可变的解剖学差异**，避免依赖可变的关节姿态
2. **浮动阈值+浮点精度=隐形炸弹**，临界值务必留安全边际
3. **归一化因子要基于真实数据分布**调节，不能拍脑袋
4. **不确定时选择最少伤害的决策**：由产品体验决定偏向 false positive 还是 false negative
5. **合成数据测试不能替代真实场景验证**，但可以快速暴露算法逻辑错误

### 踩坑 6：多模式镜像左右手标记不一致——同样的 bug 修了模式A漏了模式B

**现象**：用户反复反馈"视频窗口的左右手标记不对"，每次只修了一个模式的代码，没意识到还有其他模式存在同样问题。

**原因**：项目有 3 个视频检测模式，各自独立处理左右手标记，且 **数据源的左右交换逻辑不一致**：

| 模式 | 入口函数 | 数据源是否交换 | left_ok 真实含义 |
|------|---------|:--:|------|
| [2] 骨架预览 | `live_detection_preview` | ❌ 无交换 | 解剖学左手 |
| [3] 面部+手部 | `live_face_hands_acupoints` | ✅ 已交换 | 实际是右手（right_hand_landmarks → left_ok） |
| [4] 全身模式 | `live_full_body_acupoints` | ❌ 无交换 | 解剖学左手 |

**根因**：
1. 同一个画面镜像问题在 **3 处独立实现**，没有抽取公共函数
2. Mode [3] 在数据源层面交换了左右手数据（`right_hand_landmarks` 赋给 `hand_side="left"`），而 Mode [2]/[4] 没有交换
3. 这导致面板的 L/R 映射**不能在所有模式间统一**——Mode [3] 的面板需要交换 left_ok/right_ok，但 Mode [2]/[4] 不需要
4. 统一约定：**"L-Hand" = 用户的左手（解剖学）**，面板直接映射 `left_ok`/`right_ok`

**教训**：
- **重复逻辑必须抽取为单一函数**。镜像校正应是一个公共方法，而不是在 3 个模式中分别重复实现
- **修 bug 前先全局搜索同类模式**，确保所有调用点都被覆盖
- **定义了但未使用的变量是危险信号**——说明功能不完整或被遗漏

### 踩坑 7：错误地在 Mode [2]/[4] 也做了面板左右交换——"修反了"

**现象**：在踩坑 6 修复后，用户反馈"视频右上角的标识还是没改"，L-Hand 和 R-Hand 反倒是更不对了。

**原因**：没有理解三个模式中 `left_ok`/`right_ok` 的真实含义不同：
- Mode [2] 和 [4]：`left_ok` = 解剖学左手（数据源无交换）
- Mode [3]：`left_ok` = 解剖学右手（数据源已交换——`right_hand_landmarks` → `left_ok`）

在对 Mode [2] 和 [4] 面板加了交换（`L-Hand↔right_ok`）后，等于**双重反转**：
- 解剖右手检测到 → right_ok=True → 面板显示 L-Hand 亮 → **用户看到左手标亮起，但实际是右手**

**正确做法**：
- Mode [2]/[4] 面板：`("L-Hand", left_ok), ("R-Hand", right_ok)` —— 不交换
- Mode [3] 面板：`("L-Hand", right_ok), ("R-Hand", left_ok)` —— 因为数据源已交换
- 结果统一：**L-Hand 亮 = 解剖学左手检测到**（屏幕右侧的手）

**教训**：
- 修 bug 前必须**追踪变量在整个数据流中的含义**，不能只看局部
- 同样的变量名（`left_ok`）在不同模式中可能代表不同含义，不能假设一致
- 修完后要检查所有模式是否一致

### 踩坑 8：全身穴位偏移到身体旁边的空气中 + 手背仍显示掌心穴位

**现象 1（全身穴位偏移）**：Mode [4] 躯干穴位位置严重偏移，完全浮在身体旁边的空气中，而非贴在身体上。

**最终根因 1**（经过两次修复才找到真正的 bug）：`demo_verify.py:1625` 把 `image_shape` 设成了 `(h, w)`（高度在前），而 `pose_extractor.py` 和所有 locator 函数（`_compute_torso_2d`、`_compute_limb_2d`、`_compute_face_2d`、`_compute_hand_2d`、`_project_to_2d_improved`）都一致用 `w, h = image_shape` 解包，导致宽高互换：

- 1280×720 画面：`w=720, h=1280`（互换）
- x 坐标乘以 720 而非 1280 → 水平压缩到 56%
- y 坐标乘以 1280 而非 720 → 垂直拉伸到 178%
- 所有穴位被压向中轴并向下偏移，看起来在"身体旁边的空气中"

注意：`pose_extractor.py` 本身是正确的 `(w, h)`，只有 `demo_verify.py` Mode [4] 手动构造 `PoseResult` 时写错了。

**修复 1（最终）**：`demo_verify.py:1625` `image_shape = (h, w)` → `(w, h)`

**其他修复（辅助）**：
- 髋部可见性阈值从 0.4 → 0.15（减少误触发 fallback）
- Fallback 轴改为 `shoulder → 按比例估算髋部`（系数 1.0，之前用 1.8 偏大）
- 移除死代码 `use_fallback_axis`

**教训**：变量命名一致≠语义一致；`(h, w)` 和 `(w, h)` 是一分钟级别的 bug，但影响全局。

**现象 2（手背显示掌心穴位）**：手背朝向摄像头时，仍显示仅应在掌心呈现的穴位。

**根因 2**：
- Mode [3] `live_face_hands_acupoints` 完全跳过了 `valid_orientation` 检查
- Mode [4] `_locate_single` 做了朝向检查，但朝向不匹配时只降级为 D 级——默认过滤等级就是 D，所以仍然显示

**修复 2**：
- Mode [3]：添加 `detect_hand_orientation()` 检测，根据 `valid_orientation` 跳过不匹配的穴位
- Mode [4]：`_locate_single` 中朝向不匹配时直接 `return None`（不再降级为 D 后仍显示）
